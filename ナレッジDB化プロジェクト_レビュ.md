# ナレッジDB化プロジェクト レビュー統合版

## 1. 総評

方向性は非常に妥当です。

定年退職・ベテラン退職が近い社員から、退職前に業務知識や判断基準を収集し、DB化して残すという発想は、引継ぎ失敗による損失を防ぐうえで実務的です。

特に良い点は、単なる「手順書置き場」ではなく、検索・再利用・更新を前提とした**情報DB**として考えている点です。  
また、手順だけでなく、ベテランが持つ「判断のコツ」「失敗経験」「例外対応」まで残そうとしている点は、この取り組みの価値が高い部分です。

---

## 2. 良い点

### 2.1 課題設定が正しい

退職後に知識を集めようとしても、多くの場合は手遅れになります。  
その前に情報収集しておくという考え方は、業務継続・品質維持・再調査コスト削減の観点で正しいです。

### 2.2 対象が明確

「定年まじかの人が多い会社」という背景があり、なぜこのプロジェクトが必要なのかを説明しやすいです。  
経営層や管理職にも、業務継続リスクとして説明しやすいテーマです。

### 2.3 DB化の方向性が適切

単なるドキュメント置き場ではなく、検索・再利用・更新を前提にした情報DBとして捉えている点は適切です。

### 2.4 暗黙知まで対象にしている

通常の引継ぎでは、手順や資料の場所だけが残りがちです。  
しかし実際に困るのは、以下のような情報です。

- 例外時にどう判断するか
- 誰に確認すればよいか
- 過去に何で失敗したか
- なぜそのやり方をしているのか

ここをDB化の対象に含めている点は非常に重要です。

---

## 3. 現状の弱い点・リスク

### 3.1 情報の粒度がバラつく

人によって書き方が違うと、後から検索・比較・再利用しにくくなります。  
そのため、最初に入力テンプレートを固定する必要があります。

### 3.2 「作って終わり」になるリスク

退職前に情報を集めても、更新責任者がいなければ半年後には古くなります。  
DB化するなら、登録だけでなく更新ルールまで設計する必要があります。

### 3.3 暗黙知が抜けやすい

ベテラン本人にとって当たり前のことほど、文章に残りません。  
自由記述だけに頼ると、本当に重要な判断基準や失敗経験が抜ける可能性があります。

### 3.4 優先順位が曖昧になりやすい

全業務を一気に集めようとすると負荷が大きくなります。  
まずは「止まると困る業務」から始めるべきです。

### 3.5 責任が曖昧になりやすい

誰が更新するのか、誰が承認するのか、誰が内容の正しさを確認するのかが曖昧だと、DBが形骸化します。

---

## 4. 最初に決めるべき設計ポイント

### 4.1 収集対象の分類

最初に、集める情報を以下の4種類に分けると整理しやすくなります。

| 分類 | 内容 |
|---|---|
| 業務手順 | 定型作業、月次・年次業務、日常業務 |
| 判断基準 | 例外対応、優先順位、確認すべきポイント |
| 人脈・依存関係 | 社内外の相談先、確認先、取引先窓口 |
| 失敗知見 | 過去トラブル、やってはいけないこと、注意点 |

特に重要なのは、**判断基準・人脈・失敗知見**です。  
この3つは文書化されにくく、退職時に失われやすい情報です。

---

### 4.2 ナレッジ記録テンプレートの固定

1件ごとの記録形式は、最初から統一した方がよいです。  
最低限、以下の項目があると使いやすくなります。

```markdown
# 業務名

## 目的
この業務を行う理由・役割

## 実施タイミング
毎日 / 毎週 / 毎月 / 年次 / 随時

## 前提条件
必要な権限、資料、システム、事前準備

## 通常手順
1.
2.
3.

## 判断ポイント
迷いやすい箇所、判断基準、確認すべき数字・資料

## 例外時対応
トラブル時、イレギュラー時の対応方法

## よくある失敗
過去のミス、注意点、やってはいけないこと

## 関連資料
ファイル名、保存場所、URL、紙資料の所在

## 問い合わせ先
社内担当者、社外窓口、確認先

## 更新情報
最終更新日：
更新者：
責任者：
```

---

### 4.3 優先順位の付け方

全業務を一気にやるのではなく、次の基準で優先順位をつけるのが現実的です。

```text
優先度 = 影響度 × 消失リスク × 代替困難性
```

| 指標 | 意味 |
|---|---|
| 影響度 | 業務が止まったときの影響が大きいか |
| 消失リスク | 特定担当者しか知らないか |
| 代替困難性 | 他の人がすぐ代わりにできないか |

最初は、以下のような業務から着手すると効果が出やすいです。

- 月次・年次で必ず発生する業務
- 特定の人しか知らない業務
- 顧客・売上・法令・品質に影響する業務
- 過去にトラブルが起きた業務
- 退職予定者が担当している業務

## 5. 暗黙知を引き出す質問例

ベテラン社員には、単に「手順を書いてください」と頼むより、インタビュー形式の方が有効です。

質問例：

```text
この業務で新人が最初につまずく点はどこですか？

トラブル時に最初に確認する資料や数字は何ですか？

過去に大きな失敗があったとしたら、原因は何でしたか？

この判断は、何を基準に決めていますか？

あなたが不在のとき、後任が一番困りそうな判断は何ですか？

この業務で「これはやってはいけない」ということはありますか？

社外・社内で、困ったときに聞くべき相手は誰ですか？
```

ポイントは、**何をやるか**だけでなく、**なぜそう判断するのか**を聞くことです。

---

## 6. DB運用ルール

DBとして運用するなら、以下のルールが必要です。

| 項目 | 内容 |
|---|---|
| オーナー制 | 業務ごとに更新責任者を置く |
| 代理担当 | 責任者不在時の確認者を決める |
| 更新頻度 | 最低でも四半期に1回見直す |
| レビュー | 退職予定者・後任・管理職の3者で確認する |
| タグ設計 | 部署、業務名、システム、頻度、重要度など |
| 権限管理 | 全体公開、部門限定、機微情報に分ける |
| 完了定義 | 後任が見て実行できたら完了とする |

特に重要なのは、**後任が実際にその手順で業務を実行できるか**です。  
読めるだけではなく、再現できる状態にする必要があります。

## 7. 最小構成（MVP案）

最初から大規模に始めるより、まずは小さく試す方が現実的です。

### 7.1 初期対象

```text
対象業務：まず10件
対象部署：退職予定者が多い部署を1つ選ぶ
```

### 7.2 収集方法

```text
1件あたり：
- 60分インタビュー
- 30分レビュー
- 後任による実行確認
```

### 7.3 登録ペース

```text
週3件ずつ登録
1か月で初版作成
まず100件を中期目標にする
```

### 7.4 1件あたりの分量

```text
A4 1〜2ページ程度
長すぎず、検索しやすい形にする
```

## 8. 3か月の実行プラン

### 8.1 0〜2週目：棚卸し

- 退職予定者ごとの担当業務一覧を作成
- 業務停止時の影響度を確認
- 優先順位を決める
- テンプレートを確定する

### 8.2 3〜6週目：一次収集

- 優先度の高い業務からインタビュー
- 手順、判断基準、失敗例、連絡先を記録
- 関連資料の場所を整理

### 8.3 7〜10週目：後任による検証

- 後任がDBを見て業務を試す
- 詰まった箇所を追記
- 不足している判断基準を補う

### 8.4 11〜12週目：運用化

- 更新責任者を設定
- 月次または四半期レビューを決める
- 退職1か月前の最終確認フローを作る

---

## 9. 3年スパンのロードマップ

| 期間 | 内容 |
|---|---|
| 0〜3か月 | 対象業務の棚卸し、優先順位付け、テンプレート確定、パイロット実施 |
| 4〜12か月 | 高優先業務を中心に収集、退職予定者の知見を重点的にDB化 |
| 2年目 | 更新フロー、レビュー体制、品質確認を定着 |
| 3年目 | 新人教育、異動時引継ぎ、研修資料と連携 |

---

## 10. KPI・効果測定

成果を測る指標としては、以下が使えます。

| KPI | 内容 |
|---|---|
| 引継ぎ完了率 | 対象業務のうち、記録・確認済みの割合 |
| 後任の独力対応率 | DBを見て問い合わせなしで完了できた割合 |
| 情報更新率 | 最終更新から3か月以内の記事割合 |
| 問い合わせ削減数 | ベテラン・退職者への確認回数の減少 |
| トラブル復旧時間 | 問題発生時に解決までかかる時間 |
| 属人化業務数 | 特定担当者しかできない業務の件数 |

## 11. READMEに入れるとよい章立て

READMEを改善するなら、以下の構成がよいです。

```markdown
# ナレッジDB化プロジェクト

## 1. 目的
退職・異動により失われやすい業務知識、判断基準、失敗知見を記録し、業務継続性を高める。

## 2. 背景
定年退職予定者が多く、属人化した業務知識の消失リスクが高まっている。

## 3. 対象範囲
- 業務手順
- 判断基準
- 例外対応
- 連絡先
- 失敗事例
- 関連資料

## 4. 優先順位
影響度、消失リスク、代替困難性をもとに決定する。

## 5. 記録テンプレート
各ナレッジは共通テンプレートで記録する。

## 6. 運用ルール
更新責任者、更新頻度、レビュー方法を定める。

## 7. 完了定義
後任者が記録内容を見て業務を再現できる状態を完了とする。

## 8. 今後のロードマップ
初期パイロット、全社展開、教育連携の順に進める。
```

## 12. まず次にやるべきこと

優先順位としては、以下の3つから始めるのがよいです。

1. **退職予定者が多い部署を1つ選ぶ**
2. **ナレッジ記録テンプレートを確定する**
3. **高優先業務を10件だけ選び、パイロットで記録する**

この3つをやれば、机上の構想ではなく、実際に回る仕組みかどうかを確認できます。

---

## 13. まとめ

このプロジェクトの方向性は正しいです。  
ただし、成功させるには「情報を集める」だけでは不十分です。

重要なのは、以下の4点です。

```text
何を優先して残すか
どの型で記録するか
誰が更新するか
後任が本当に使えるか
```

最初から完璧なDBを作るより、まずは10件程度の業務で小さく試し、後任が実際に使える形に修正していくのが現実的です。
