# 業務知識DB・属人化対策サービスを提供する企業候補まとめ

作成日：2026-05-15  
用途：上司説明用・業者比較のたたき台  
目的：社員の属人知識、暗黙知、過去案件、判断基準を会社の知識資産として整理・検索・活用するために、外部サービス／支援企業の候補を整理する。

---

## 1. 結論

社員の属人的な知識をDB化し、必要に応じてAI検索・RAG・社内チャットボットで活用するサービスは、すでに複数存在する。

ただし、候補企業は一括りにしない方がよい。大きく以下の4種類に分けて検討する。

| 分類 | 主な役割 | 代表候補 |
|---|---|---|
| 大手SI・コンサル系 | 構想策定、RAG基盤、権限設計、ガバナンス、全社展開 | NTTデータ、日立、PwC、アクセンチュア、IBMなど |
| ナレッジDB・社内Wiki系 | 手順書、FAQ、ノウハウ、社内知識を蓄積・検索 | Qast、NotePM、Helpfeel、Confluence、Notionなど |
| 業務DB・ノーコード系 | 業務台帳、案件管理、顧客注意点、トラブル事例のDB化 | kintone、Microsoft Lists、Power Appsなど |
| 既存ファイル・AI活用系 | SharePoint、Boxなど既存資料をAI検索・要約・抽出に使う | Microsoft Copilot Studio、Box AI、exaBase生成AI、LayerX Ai Workforce、ELYZA Worksなど |

今回の構想では、**「まず小さく業務知識を洗い出す」→「既存ツールで試作」→「必要なら専門業者に相談」** が現実的である。

## 2. 候補企業・サービス一覧

## 2-1. 大手SI・コンサル系

### NTTデータ

#### 概要
NTTデータは、RAG構築、AIプラットフォーム導入、AIリテラシー向上、ROI創出、データ蓄積・加工基盤、データマネジメント、運用・ガバナンスの支援などを扱っている。

#### 今回の構想との相性
全社的に業務知識DB、RAG基盤、AI活用基盤を構築したい場合に向く。

#### 向いている用途
- 全社・部門横断のRAG基盤構築
- 社内データ活用基盤
- AIガバナンス設計
- データマネジメント体制の整備
- 将来的なAIエージェント活用

#### 留意点
初期段階の小さな試作より、会社として本格的に予算をかけて進める段階で相談する方が向いている。

#### 参考
- NTTデータ「生成AI活用におけるRAG導入のポイント」  
  https://www.nttdata.com/jp/ja/trends/data-insight/2025/0303/

### 日立グループ

#### 概要
日立は、熟練者のノウハウを取り込んだ業務特化型AIエージェントや、製造業・設備管理向けのAI活用を打ち出している。日立ソリューションズも、現場向けナレッジ蓄積・継承ソリューションを提供している。

#### 今回の構想との相性
製造、設備、保守、技術問い合わせ、現場判断など、熟練者の暗黙知が重要な領域と相性がよい。

#### 向いている用途
- 設備保全ノウハウの継承
- 保守問い合わせ対応AI
- 製造業向けアシスタントAI
- 熟練者の判断・経験則の形式知化
- 現場作業者向けのナレッジ提示

#### 留意点
営業・事務・案件管理系の知識DBにも応用可能だが、特に強みが出るのは製造・設備・保守などの現場知識領域である。

### PwCコンサルティング

#### 概要
PwCは、社内規程、業務プロセス、自社用語体系に対応したRAG共通基盤の構想・設計・構築を支援している。加えて、導入後に使われる仕組みにするため、ユーザー調査や現場定着を重視している。

#### 今回の構想との相性
経営層向けに、全社的なRAG基盤・ナレッジ活用基盤として企画する場合に向く。

#### 向いている用途
- 全社RAG基盤の構想策定
- ナレッジ活用基盤の設計
- セキュリティリスクを抑えた生成AI利用環境
- 現場に定着する仕組み作り
- 経営層向けの構想・ロードマップ作成

#### 留意点
実務部門の小さな資料整理から始める段階では、やや大きめの選択肢。正式な全社施策にする段階で有効。

## 2-2. ナレッジDB・社内Wiki・AI検索系

### Qast

#### 概要
Qastは、AIナレッジプラットフォームとして、蓄積されたナレッジをもとにAIが回答を生成するRAG機能や、AIの質問に声で答えることで属人化したノウハウを構造化されたナレッジに変換する「AIナレッジインタビュー」を提供している。

#### 今回の構想との相性
今回の「ベテランの暗黙知を聞き取り、業務知識DB化したい」という目的にかなり近い。

#### 向いている用途
- 属人化したノウハウの形式知化
- ベテラン社員へのインタビュー
- 社内FAQ・ナレッジDB
- RAGによる社内知識検索
- 回答ソースの確認

#### 留意点
業務フローや案件管理を細かくDB化する場合は、kintoneやMicrosoft Listsなどとの併用も検討した方がよい。

#### 参考
- Qast「機能」  
  https://qast.jp/function/

---

### NotePM

#### 概要
NotePMは、マニュアル作成、社内ポータル、ノウハウ共有、ドキュメント管理などに使えるナレッジ管理サービスである。AI検索機能では、蓄積された社内ナレッジの中からAIが情報を探し、要約して回答する。

#### 今回の構想との相性
まず手順書、FAQ、業務メモ、社内Wikiを整備したい場合に向いている。

#### 向いている用途
- 手順書・マニュアル管理
- 社内FAQ
- 引継ぎ資料管理
- ノウハウ共有
- AIチャットボットによる社内ナレッジ検索

#### 留意点
「案件履歴DB」「顧客注意点DB」のような構造化データ管理は、別途kintoneやSharePointリストなどと組み合わせるとよい。

#### 参考
- NotePM公式サイト  
  https://notepm.jp/
- NotePM「AI検索（チャットボット）機能」  
  https://help.notepm.jp/hc/ja/articles/54228006776729-AI%E6%A4%9C%E7%B4%A2-%E3%83%81%E3%83%A3%E3%83%83%E3%83%88%E3%83%9C%E3%83%83%E3%83%88-%E6%A9%9F%E8%83%BD

### Helpfeel

#### 概要
Helpfeelは、社内ナレッジ管理・検索、FAQ、ヘルプデスク支援に強いサービスである。AIと検索技術により、社内ナレッジを見つけやすくすることを打ち出している。

#### 今回の構想との相性
「社員が質問したときに、社内FAQや資料からすぐ答えを探せる状態」を作りたい場合に向く。

#### 向いている用途
- 社内問い合わせ削減
- FAQ検索
- ヘルプデスク業務支援
- 社内ナレッジ検索
- 顧客対応・問い合わせ対応の高速化

#### 留意点
業務知識そのものを一から構造化するというより、既存のFAQやナレッジを探しやすくする用途に強い。

#### 参考
- Helpfeel「社内ナレッジがすぐ見つかる AIヘルプデスク」

## 2-3. 業務DB・ノーコード系

### kintone

#### 概要
kintoneは、プログラミングの知識がなくても業務アプリを作れるノーコードツールである。顧客管理、案件管理、クレーム管理、見積管理、契約書管理、社内FAQなどの業務アプリ作成例がある。

#### 今回の構想との相性
「業務知識を構造化して台帳化する」用途にかなり向いている。

#### 向いている用途
- 業務一覧DB
- 案件履歴DB
- 顧客・取引先注意点DB
- トラブル事例DB
- 見積管理DB
- 引継ぎ状況管理
- アンケート回答の集計先

#### 留意点
文章中心のマニュアルや社内WikiにはNotePMやConfluenceの方が向く場合がある。kintoneは「項目を決めて管理するDB」として使うと強い。

#### 参考
- kintone公式サイト  
  https://kintone.cybozu.co.jp/
- kintone「ノーコードでDXを進めよう」  
  https://kintone.cybozu.co.jp/jp/kintone-dx/

## 2-4. Microsoft 365・Boxなど既存ファイル活用系

### Microsoft Copilot Studio / SharePoint / Microsoft Lists

#### 概要
Microsoft Copilot Studioでは、SharePointサイトやSharePointリストをエージェントのナレッジソースとして追加できる。SharePointの権限を使い、ユーザーがアクセス権を持つ情報をもとに回答する仕組みを作れる。

#### 今回の構想との相性
会社がMicrosoft 365を使っている場合、最初に検討すべき候補。

#### 向いている用途
- SharePoint上の資料をAI検索対象にする
- SharePointリストを知識DBとして使う
- Microsoft Listsで業務台帳を作る
- Copilot Studioで社内向けエージェントを作る
- 既存Microsoft 365権限を活かした運用

#### 留意点
便利だが、元資料の整理・命名ルール・更新ルールが弱いと、AI検索の精度も落ちる。先に情報整理が必要。

#### 参考
- Microsoft Learn「ナレッジ ソースとしてSharePointを追加する」  
  https://learn.microsoft.com/ja-jp/microsoft-copilot-studio/knowledge-add-sharepoint
- Microsoft Learn「Knowledge sources summary」  
  https://learn.microsoft.com/en-us/microsoft-copilot-studio/knowledge-copilot-studio

---

### Box AI

#### 概要
Box AIは、Box上の文書に対する質問、要約、抽出などに使えるAI機能を提供している。Box Extractでは、文書から必要なデータを抽出する用途もある。

#### 今回の構想との相性
会社がBoxを中心に資料管理している場合、既存資料を活かしやすい。

#### 向いている用途
- Box内文書の要約
- Box内ファイルへの質問回答
- 契約書、申請書、帳票などからの情報抽出
- 既存ファイル資産のAI活用
- Boxの権限管理を活かした資料活用

#### 留意点
Box内のフォルダ構成やファイル名、最新版管理が乱れている場合は、AI以前に整理ルールが必要である。Box AIは「資料を整理してくれる魔法」ではなく、整理された資料を活用しやすくする手段として考えるべき。

#### 参考
- Box Developer「Ask questions to Box AI」  
  https://github.com/box/developer.box.com/blob/main/content/guides/box-ai/ai-tutorials/ask-questions.md
- Box「Box Extract」  
  https://www.box.com/extract

- 提案書作成や契約書レビューなどのドキュメント業務
- 社内DB・外部データソースを参照するAI活用
- エンタープライズ向けAI基盤

#### 留意点
初期の業務洗い出し段階ではやや大きい可能性がある。業務知識DBを作った後、AIエージェント化・業務自動化まで進める段階で検討したい。

### ELYZA Works

ELYZA Worksは、法人向け生成AI活用ツールで、自社専用のAIアプリを現場主導で作成し、チームで共有・改善できるサービスである。RAG機能等にも対応し、自社固有の情報を参照させたAIアプリ作成も可能と説明されている。

#### 今回の構想との相性
部門ごとに「見積補助」「問い合わせ回答」「資料作成補助」などのAIアプリを作りたい場合に向く。

#### 向いている用途

- RAGを使った自社固有情報の参照
- 入力項目が明確な業務AIアプリ
- 業務改善アプリの試作

#### 留意点
ナレッジDBそのものを整備するというより、整備済みの知識や資料を使って業務AIアプリ化する用途で考えるとよい。

## 3. 今回の構想に対するおすすめ分類

## 小さく始める場合

まずは、以下の候補が現実的である。

| 候補 | 理由 |
|---|---|
| SharePoint / Microsoft Lists | 会社がMicrosoft 365を使っている場合、既存権限を活かしやすい |
| kintone | 業務一覧、案件履歴、顧客注意点、トラブル事例をDB化しやすい |
| NotePM | 手順書、FAQ、マニュアル、社内Wikiを作りやすい |
| Qast | 属人化したノウハウや暗黙知の形式知化に近い |
| Box AI | Boxに資料が多い場合、既存資料のAI活用候補になる |

---

## 本格導入する場合

全社基盤、セキュリティ、権限、AI/RAG構築、ガバナンスまで考えるなら、以下が候補になる。

| 候補 | 理由 |
|---|---|
| NTTデータ | RAG、データ基盤、AIガバナンス、伴走支援まで広い |
| 日立 | 製造・設備・保守・熟練者ノウハウのAI活用に強い |
| PwC | 全社RAG基盤、構想・設計・現場定着支援に向く |
| アクセンチュア / IBM / デロイト | 大規模DX、生成AI導入、業務改革込みなら候補 |

## 4. サービス選定の考え方

業者を選ぶ前に、まず自社側で以下を整理する必要がある。

| 整理項目 | 内容 |
|---|---|
| 目的 | 退職対策、引継ぎ、教育、営業力向上、問い合わせ削減など |
| 対象業務 | どの業務をDB化するのか |
| 対象知識 | 手順、判断基準、過去案件、顧客注意点、トラブル事例など |
| 既存資料 | Box、SharePoint、個人フォルダ、メール、紙資料など |
| 利用者 | 若手、後任者、営業、管理職、技術担当など |
| 利用場面 | 引継ぎ、新人教育、案件対応、問い合わせ、トラブル対応など |
| 権限 | 誰がどの情報を見られるか |
| 更新ルール | 誰が更新し、誰が承認するか |
| AI利用範囲 | 検索だけか、要約・回答・文書作成まで行うか |

## 5. 業者に確認すべき質問

外部業者に相談する際は、以下を確認する。

```text
1. 属人化した業務知識をどのように洗い出すか
2. ベテラン社員へのヒアリングや暗黙知抽出まで支援できるか
3. 既存のBox / SharePoint / Excel / PDFを活用できるか
4. 判断基準・過去経緯・トラブル事例をどのようにDB化できるか
5. AI検索やRAGを導入する場合、回答根拠を表示できるか
6. 閲覧権限・機密情報・顧客情報をどう制御するか
7. 古い資料や誤情報をどう扱うか
8. 更新責任者やレビュー運用をどう設計するか
9. 小規模PoCから始められるか
10. 導入後に現場で使われるようにする支援があるか
```

## 7. 今回のおすすめ方針

### 第1段階：自社側で整理する

まずは、外部業者に相談する前に以下を行う。

```text
1. 業務知識・引継ぎ状況アンケートを実施する
2. 属人化している業務を洗い出す
3. 優先度の高い業務を3〜5件選ぶ
4. 既存資料の有無を確認する
5. 資料にない判断基準・注意点をヒアリングする
```

### 第2段階：小さく試作する

いきなり大規模導入せず、既存ツールで試作する。

```text
候補：Excel / SharePoint / Microsoft Lists / kintone / NotePM / Qast
対象：重要業務3〜5件
目的：本当にDB化すべき知識が何かを確認する
```

### 第3段階：業者に相談する

試作結果をもとに、以下のような相談を行う。

```text
・この業務知識をどのようなDB構成にすべきか
・BoxやSharePointの既存資料を活用できるか
・AI検索やRAGを導入する場合の費用感はどの程度か
・権限管理やセキュリティをどう設計すべきか
・部内PoCから全社展開へ進める場合のロードマップはどうなるか
```

---

## 8. 上司向け説明文

> このような属人知識・業務知識のDB化やAI検索は、すでに複数の企業がサービスとして提供しています。  
> ただし、最初から外部業者に丸投げするのではなく、まず当社側で「どの業務の、どの知識が失われると困るのか」を整理する必要があります。  
> そのうえで、小規模に業務知識DBを試作し、必要に応じてQast、NotePM、kintone、Microsoft Copilot Studio、NTTデータ、日立、PwC等のサービス・支援を比較検討するのが現実的です。

## 9. 一言でのまとめ

この領域のサービスは存在する。  
ただし、重要なのは「どのサービスを入れるか」より先に、**自社として何の知識を残すべきかを決めること**である。

最初のゴールは、完璧なAIシステムではなく、以下である。

```text
退職・異動・長期不在が発生しても、
重要業務について、手順・判断基準・過去経緯・注意点を確認できる状態を作る。
```

その後、必要に応じて外部サービスや専門業者を活用し、業務知識DB・AI検索・RAG基盤へ発展させる。
