# 業務知識DB構想・実行計画 統合版

## 0. この文書の位置付け

この文書は、以下2つの内容を統合したものである。

- **DB立案の意図**：なぜ業務知識DBが必要なのか、何を保存すべきか、最終的にどう使いたいのか
- **レビュー統合版**：どの順番で進めるか、どう記録するか、誰が更新するか、どう成果を測るか

つまり、本書は単なるアイデア整理ではなく、**社内説明・初期提案・実行計画の土台として使うための統合版**である。

---

## 1. 目的

社員の頭の中にある業務知識、属人的な判断、過去の経緯、顧客対応の注意点などを会社の資産として保存し、退職・異動・定年によって業務が止まるリスクを減らす。

この取り組みの本質は、単なるマニュアル作成ではない。

**「人の頭の中にある業務の理由・判断・経験を、会社全体で使える知識資産に変えること」** が目的である。

---

## 2. 背景・課題

現在のように、特定の社員に業務知識が集中している状態では、以下のリスクがある。

- 退職・定年で知識が失われる
- 異動時に引き継ぎが追いつかない
- 若手が過去の経緯を知らず、同じ失敗を繰り返す
- 判断基準が人によって違う
- 「詳しい人に聞かないと分からない」状態が続く
- 管理職やベテランが一斉に退職した際、部門全体の判断力が落ちる
- 客先対応や過去案件の注意点が継承されない

特に大きな問題は、手順そのものよりも、**判断基準や過去経緯が失われること**である。

---

## 3. 結論

やるべきことは、広い意味では **業務知識DBの作成** である。

ただし、単なるファイル置き場やExcel管理では不十分である。以下を含む、検索・再利用・更新を前提にした業務知識DBとして整備する必要がある。

- 業務手順
- 判断基準
- 例外対応
- 過去トラブル
- 顧客・取引先ごとの注意点
- 関係者・確認先
- 過去の成果物
- なぜ今の運用になっているのかという経緯
- ベテラン社員の経験則、暗黙知

最終的なイメージは以下である。

```text
社内版Wikipedia
＋
過去案件検索システム
＋
ベテランの判断基準集
＋
AIに聞ける業務相談窓口
```

---

## 4. この取り組みで重要な考え方

### 4.1 手順書ではなく、判断を残す

通常の引継ぎでは、手順や資料の場所だけが残りがちである。

しかし実際に困るのは、以下のような情報である。

- 例外時にどう判断するか
- 誰に確認すればよいか
- 過去に何で失敗したか
- なぜそのやり方をしているのか
- この客先では何に注意すべきか

そのため、このDBでは単なる作業手順だけでなく、**判断基準・例外対応・過去経緯・失敗知見**を重点的に残す。

### 4.2 完璧ではなく、80点で再現できる状態を目指す

100点を目指してすべてを完璧に文書化しようとすると、作業量が大きくなりすぎて失敗しやすい。

最初は、担当者が変わっても別の社員が**80点レベルで業務を再現できる状態**を目指す。

### 4.3 最初から大規模にやらない

全業務を一気にDB化しようとすると、負荷が大きくなり、途中で止まりやすい。

まずは、退職したら困る業務、過去トラブルが多い業務、客先影響が大きい業務など、危険度の高い業務から小さく始める。

### 4.4 現時点で優先して潰すべき指摘・リスク

この構想は方向性としては妥当だが、実行段階では以下のリスクを先に潰しておく必要がある。  
ここは、社内レビューや上司説明で指摘されやすい論点でもある。

| 指摘・リスク | 具体的に起きる問題 | 先に決めるべき対策 |
|---|---|---|
| 作って終わり化 | 更新責任者・レビュー頻度・承認フローが弱いと、半年程度で情報が陳腐化する | 業務ごとに更新責任者、承認者、代理担当、レビュー頻度、未更新時の是正ルールを固定する |
| 記述品質のバラつき | 自由記述中心だと、人によって粒度や表現が変わり、検索性・再利用性が落ちる | 1業務1ページのテンプレート、必須項目、記入例、タグ、表記ルールを固定する |
| 優先度の主観化 | 「重要そう」という感覚で始めると、工数配分が偏り、重要業務が後回しになる | 優先度 = 影響度 × 消失リスク × 代替困難性でスコア化する |
| 暗黙知の抜け漏れ | ベテラン本人は“当たり前”を省略しがちで、重要な判断基準や注意点が残らない | インタビュー設問を固定し、判断・例外・失敗・過去経緯を必ず聞く |
| 成果指標の未定義 | 「良くなったか」を定量で示せず、上司・経営層への継続説明が難しくなる | 検索時間、問い合わせ件数、引継ぎ完了率、更新遵守率などのKPIを先に決める |
| 情報設計の深さ不足 | 顧客別注意点、判断基準、トラブル事例は分類軸を誤ると検索不能になりやすい | 業務名、顧客、システム、頻度、重要度、情報種別、公開範囲などのタグを先に決める |
| 登録は進むが使われない | 利用シーンが未定義だと、登録だけ進んでも検索されないDBになる | 誰が、いつ、どの判断で使うのかという利用導線を設計する |
| ツール選定先行 | ベンダー比較や製品選定が先行すると、入力設計・運用設計が弱いままシステム化される | まず入力テンプレート、運用ルール、MVPを固め、不足機能を確認してからツールを選ぶ |

特に重要なのは、**DBを作ること自体を目的にしないこと**である。  
目的は、退職・異動後も別の社員が業務を再現でき、必要な判断材料を検索できる状態を作ることである。

### 4.5 指摘を運用設計へ落とす考え方

上記の指摘は、単なる注意点として書くだけでは不十分である。  
以下のように、実際の運用項目へ落とし込む必要がある。

| 指摘 | 運用へ落とす項目 |
|---|---|
| 作って終わり化 | 更新責任者、承認者、レビュー頻度、未更新時の通知・是正 |
| 記述品質のバラつき | テンプレート、記入例、必須項目、レビュー観点 |
| 優先度の主観化 | 優先度スコア、属人化業務一覧、着手順の決定ルール |
| 暗黙知の抜け漏れ | 聞き取りシート、固定質問、後任による実行検証 |
| 成果指標の未定義 | KPI、月次確認、上司報告用の集計項目 |
| 情報設計の深さ不足 | タグ設計、分類軸、検索項目、命名ルール |
| 登録は進むが使われない | 利用シーン、検索導線、教育・研修への組込み |
| ツール選定先行 | MVP検証、必要機能一覧、ツール選定基準 |

## 5. 保存すべき知識の種類

| 種類 | 内容 | 保存方法 |
|---|---|---|
| 手順知識 | どの順番で作業するか | 手順書、チェックリスト、動画 |
| 判断知識 | AとBのどちらを選ぶか、なぜそうするか | 判断基準表、ケース集 |
| 例外対応 | 通常と違うケースでどう処理するか | FAQ、トラブル対応DB |
| 顧客・取引先知識 | 客先ごとの癖、注意点、過去経緯 | 顧客別メモ、案件履歴 |
| 人脈・確認先 | 誰に聞けば早いか、どの部署を通すべきか | 関係者マップ |
| 過去経緯 | なぜ今のルール・運用になったか | 決定ログ、議事録、経緯メモ |
| 暗黙知 | ベテランの勘、経験則、注意点 | インタビュー、OJT記録、ケースレビュー |
| 成果物知識 | 過去の見積書、提案書、報告書、テンプレート | 成果物DB、ファイル保管庫 |

特に残すべき重要情報は、以下の5つである。

```text
1. なぜそうするのか
2. 何を見て判断するのか
3. どこで失敗しやすいのか
4. 誰に確認すべきなのか
5. 過去に何があったのか
```

この5つが残っていれば、担当者が変わっても業務を再現できる可能性が高くなる。

## 6. 業務知識DBの構成案

業務知識DBは、以下の分類で整理すると使いやすい。

### 6.1 業務手順DB

業務ごとの基本的な手順を保存する。

例：

- 見積作成手順
- 客先提出資料の作成手順
- 社内承認フロー
- システム入力手順
- 月次処理
- 問い合わせ対応

### 6.2 判断基準DB

担当者の経験に依存している判断を保存する。

例：

- この条件ならAの処理を選ぶ
- この金額以上なら上長確認が必要
- この客先の場合は事前に電話確認する
- この案件では過去に問題があったため注意する
- この資料は社外提出前に特定部署へ確認する

### 6.3 トラブル事例DB

過去に起きた失敗やクレーム、再発防止策を保存する。

例：

- 過去の納期遅延
- 見積条件の認識違い
- 客先提出資料のミス
- 社内確認漏れ
- システム入力ミス
- 類似案件での注意点

### 6.4 顧客・取引先注意点DB

客先や取引先ごとの注意事項を保存する。

例：

- この客先はメールより電話の方が早い
- この担当者は資料形式に厳しい
- この客先は納期回答を早めに求める
- 過去にこの条件で揉めたことがある
- この取引先は回答が遅れやすい

### 6.5 人材・スキルDB

社内で誰が何に詳しいかを整理する。

例：

- Aさん：見積条件に詳しい
- Bさん：特定客先に詳しい
- Cさん：システム処理に詳しい
- Dさん：過去案件の経緯を知っている

これにより、「誰に聞けばよいか」が明確になる。

### 6.6 成果物DB

過去の資料や成果物を検索できるようにする。

例：

- 過去の見積書
- 提案書
- 報告書
- 契約関連資料
- 客先提出資料
- テンプレート
- 過去案件の完成ファイル

### 6.7 決定経緯DB

なぜ現在のルールや運用になったのかを保存する。

例：

- この運用になった理由
- 過去に廃止された方法
- ルール変更の背景
- 会議での決定内容
- トラブル後に追加された確認手順

これがないと、後任者が過去の事情を知らずに同じ問題を繰り返す可能性がある。

## 7. 優先順位の付け方

全業務を一度に整備するのではなく、危険度の高いものから着手する。

優先順位は、以下の考え方で決める。

```text
優先度 = 影響度 × 消失リスク × 代替困難性
```

| 指標 | 意味 |
|---|---|
| 影響度 | 業務が止まったときの影響が大きいか |
| 消失リスク | 特定担当者しか知らないか |
| 代替困難性 | 他の人がすぐ代わりにできないか |

優先すべき業務は、以下である。

```text
退職したら止まる業務
異動したら困る業務
管理職しか判断できない業務
過去トラブルが多い業務
客先影響が大きい業務
月次・年次で必ず発生する業務
```

## 8. 最初に作るべき管理表

まずは全業務を対象に、属人化リスクを可視化する。

| 業務名 | 現担当者 | 代替可能者 | 属人化度 | 失われた場合の影響 | 優先度 | 備考 |
|---|---|---|---|---|---|---|
| 見積作成 | Aさん | Bさん | 高 | 大 | 最優先 | 判断基準が属人化 |
| 客先対応 | Cさん | なし | 高 | 大 | 最優先 | 過去経緯をCさんしか知らない |
| 月次処理 | Dさん | Eさん | 中 | 中 | 高 | 手順書はあるが古い |
| システム入力 | Fさん | なし | 高 | 中 | 高 | 例外処理が未整理 |

この管理表によって、どの業務から着手すべきかを判断しやすくなる。

## 9. 1業務1ページのテンプレート

業務ごとに、以下の形式で1ページにまとめる。

```markdown
# 業務名

## 目的
この業務を行う理由・役割

## 実施タイミング
毎日 / 毎週 / 毎月 / 年次 / 随時

## 前提条件
必要な権限、資料、システム、事前準備

## 使用する資料・システム
- 使用資料：
- 使用システム：
- 保存場所：

## 関係部署・関係者
- 社内関係者：
- 社外関係者：
- 確認先：

## 基本手順
1.
2.
3.

## 判断基準
迷いやすい箇所、判断基準、確認すべき数字・資料

## 例外パターン・例外時対応
通常と異なるケース、トラブル時、イレギュラー時の対応方法

## 過去の失敗例・よくあるミス
過去のミス、注意点、やってはいけないこと

## ベテランの注意点
資料には書かれていないが、実務上気をつけていること

## 最終成果物のサンプル
見積書、報告書、提出資料などの完成例

## 関連資料リンク
ファイル名、保存場所、URL、紙資料の所在

## 更新情報
最終更新日：
更新者：
責任者：
次回見直し日：
代替担当者：
```

重要なのは、手順だけでなく **判断基準・例外・過去失敗・注意点** を同時に残すことである。

### 9.1 記述標準化のためのルール

自由記述だけにすると、人によって粒度や表現がばらつき、後から検索・再利用しづらくなる。そのため、以下のルールを固定する。

| ルール | 内容 |
|---|---|
| 必須項目を固定する | 目的、手順、判断基準、例外対応、失敗例、関連資料、更新情報は必ず記入する |
| 不明点を空欄にしない | 分からない場合は「不明」「要確認」と明記する |
| 判断理由を書く | 「何をするか」だけでなく「なぜそうするか」を書く |
| 1記事1業務にする | 複数業務を1ページに混ぜない |
| タグを必ず付ける | 部署、業務名、頻度、重要度、情報種別、公開範囲を付ける |
| 更新情報を残す | 最終更新日、更新者、責任者、次回見直し日を記録する |

## 10. ベテラン社員への聞き取り項目

ベテラン社員には、手順書を書いてもらうより、まずインタビューする方がよい。

自由記述にすると、本人にとって当たり前すぎる重要情報が抜けやすい。質問形式で、判断・失敗・例外・経緯を引き出す。

質問例：

```text
この業務で新人が一番ミスるところは？
過去に一番大きく揉めた案件は？
この業務で絶対に確認すべき相手は？
資料には書いていないが、実際には気をつけていることは？
この判断を間違えると何が起きる？
あなたが退職したら一番困る業務は？
この業務で昔と今で変わった点は？
なぜ今のやり方になったのか？
この客先・取引先で注意すべきことは？
過去資料を見るならどのフォルダ・どの案件を見るべきか？
トラブル時に最初に見る数字・資料は？
後任が一番困りそうな判断は？
これはやってはいけない、ということは？
```

この聞き取り内容を録音・文字起こしし、AIで初期整理すると効率がよい。

## 11. 導入ステップ

### Step 1：対象業務の棚卸し

まず、業務一覧を作成する。

- どの業務があるか
- 誰が担当しているか
- 代替できる人がいるか
- 属人化しているか
- 失われた場合の影響はどれくらいか

### Step 2：優先順位を決める

全業務を一度にやらず、危険度の高いものから着手する。

優先すべき業務：

- 退職予定者が担当している業務
- 管理職しか知らない業務
- 客先影響が大きい業務
- トラブル時に大きな損失が出る業務
- 若手が引き継ぎにくい業務

### Step 3：聞き取りを行う

ベテラン社員や担当者から、手順だけでなく判断基準や過去経緯を聞き取る。

### Step 4：1業務1ページに整理する

聞き取った内容をテンプレートに沿って整理する。

最初から完璧を目指さず、80点でよい。

### Step 5：実案件で検証する

別の社員が、その資料だけを見て実際に業務を進められるか確認する。

詰まった箇所が、知識の抜けである。

### Step 6：更新ルールを決める

DBは作って終わりではない。

以下を決める必要がある。

- 誰が更新するか
- いつ見直すか
- 古い情報をどう扱うか
- 承認済み情報とメモ情報をどう分けるか
- 間違った情報を誰が修正するか

### Step 7：AI検索を導入する

資料がある程度整理されてから、AI検索やRAGを導入する。

先にAIを入れても、元資料が汚ければ役に立たない。

## 12. MVP案：最小構成で始める

最初から大規模システムを作る必要はない。

まずは小さく始め、実際に使えるかを確認する。

### 初期対象

```text
対象部署：退職予定者が多い部署を1つ選ぶ
対象業務：まず10件
対象者：退職予定者、ベテラン社員、業務が属人化している担当者
```

### 収集方法

```text
1件あたり：
- 60分インタビュー
- 30分レビュー
- 後任による実行確認
```

### 登録ペース

```text
週3件ずつ登録
1か月で初版作成
まず100件を中期目標にする
```

### 1件あたりの分量

```text
A4 1〜2ページ程度
長すぎず、検索しやすい形にする
```

## 13. 運用ルール

業務知識DBとして運用するなら、登録だけでなく更新・承認・検索のルールが必要である。ここが曖昧だと、DBは作って終わりになり、数か月後には古い情報の置き場になってしまう。

| 項目 | 内容 |
|---|---|
| 更新責任者 | 業務ごとに、記事を更新する担当者を1名決める |
| 承認者 | 業務内容の正しさを確認する管理職または有識者を決める |
| 代理担当 | 責任者不在時に確認・更新できる人を決める |
| 更新頻度 | 最低でも四半期に1回見直す。重要業務は月次でもよい |
| 承認フロー | 下書き → レビュー → 承認済み → 定期見直し、の状態を分ける |
| レビュー | 退職予定者・後任・管理職の3者で確認する |
| タグ設計 | 部署、業務名、システム、頻度、重要度などを統一する |
| 権限管理 | 全体公開、部門限定、機微情報に分ける |
| 改訂履歴 | いつ、誰が、何を変更したかを残す |
| 完了定義 | 後任が見て実行できたら完了とする |

### 13.1 承認フローの考え方

最初から厳密なワークフローシステムを作る必要はないが、最低限、以下の状態は分ける。

```text
下書き：聞き取り内容を整理した段階。未確認情報を含む。
レビュー中：後任者・管理職・有識者が内容を確認している段階。
承認済み：業務手順として参照してよい状態。
要更新：内容が古い、または実務とズレている可能性がある状態。
廃止：現在は使わないが、過去経緯として残す状態。
```

特に重要なのは、**後任が実際にその手順で業務を実行できるか**である。

読めるだけではなく、再現できる状態にする必要がある。

### 13.2 利用導線の設計

DBは、登録されているだけでは使われない。  
誰が、いつ、どの場面で参照するのかを先に決め、実務の流れに組み込む必要がある。

| 利用場面 | 使う人 | 参照する情報 |
|---|---|---|
| 退職・異動時の引継ぎ | 後任者、管理職 | 業務手順、判断基準、関連資料、確認先 |
| 新人・若手教育 | 新人、教育担当 | 基本手順、よくある失敗、成果物サンプル |
| トラブル発生時 | 担当者、管理職 | 過去トラブル事例、例外対応、確認先 |
| 顧客対応前 | 営業、担当者 | 顧客別注意点、過去経緯、類似案件 |
| 業務改善時 | 管理職、担当者 | 決定経緯、過去の廃止ルール、失敗事例 |

利用導線が弱いと、情報は登録されても現場で検索されず、結局「詳しい人に聞く」運用に戻る。  
そのため、DBを作った後ではなく、**作る前に利用場面を決める**必要がある。

## 14. 検索性とタグ設計

DB化する以上、検索できることが重要である。

記事や資料には、最低限以下のタグを付ける。

| タグ | 例 |
|---|---|
| 部署 | 営業、技術、総務、経理 |
| 業務名 | 見積作成、客先対応、月次処理 |
| システム | 社内システム名、Excel、Box、SharePoint |
| 頻度 | 毎日、毎月、年次、随時 |
| 重要度 | 高、中、低 |
| 属人化度 | 高、中、低 |
| 顧客・取引先 | 顧客名、取引先名 |
| 情報種別 | 手順、判断基準、トラブル、成果物、経緯 |
| 公開範囲 | 全体公開、部門限定、機微情報 |

資料名やファイル名だけで管理すると、後から探せなくなる。

### 14.1 情報設計で先に決める分類軸

顧客別注意点、判断基準、トラブル事例は、情報量が増えるほど検索が難しくなる。  
そのため、登録前に分類軸を決めておく。

| 情報種別 | 追加で持たせたい分類軸 |
|---|---|
| 顧客・取引先注意点 | 顧客名、担当者名、地域、案件種別、注意事項の種類、最終確認日 |
| 判断基準 | 判断対象、判断条件、確認資料、承認者、金額・期限などの基準値 |
| トラブル事例 | 発生日、原因、影響範囲、再発防止策、類似案件、関連顧客 |
| 成果物 | 案件名、年度、顧客名、資料種別、完成版・途中版の区分 |
| 決定経緯 | 決定日、決定者、変更理由、廃止された旧ルール、関連会議 |

この分類軸がないと、資料数が増えた段階で「登録はされているが探せない」状態になる。  
特に、顧客別注意点・判断基準・トラブル事例は、最初にタグ設計をしておく必要がある。

## 15. AI活用方針

AIは、知識保存の中心ではなく、**整理・検索・初期作成の補助**として使う。

### AIに向いていること

- 録音したインタビューの文字起こし
- 長い説明の要約
- 手順書の初版作成
- FAQ化
- 過去資料の分類
- 類似案件の検索
- 資料から注意点を抽出
- 質問に対して関連資料を提示する
- 新人向け説明文の作成

### AIに任せきりにしてはいけないこと

- 最終的な業務判断
- 社内ルールの確定
- 法務・契約・金額判断
- 顧客対応方針の決定
- 古い資料と新しい資料の正誤判定

AIは便利だが、元データが間違っていれば回答も間違う。

そのため、AI導入前に **資料の整理・更新ルール・責任者設定** が必要である。

## 16. 推奨する保存先・ツール構成

最初から大規模システムを作る必要はない。

また、ツール選定を先に進めすぎないことが重要である。ベンダー比較やシステム導入が先行すると、肝心の入力設計・更新責任・承認フロー・タグ設計が弱くなり、結果として使われないDBになる可能性がある。

順番としては、以下が望ましい。

```text
1. 何を残すかを決める
2. どのテンプレートで記録するかを決める
3. 誰が更新・承認するかを決める
4. どう検索するか、タグを決める
5. その運用に合うツールを選ぶ
```

### 小規模開始

- Excel
- SharePoint
- Microsoft Lists
- 社内共有フォルダ
- OneNote
- Notion
- Confluence

### 中規模以上

- SharePoint + Microsoft Lists
- kintone
- Confluence
- Box + メタデータ管理
- 社内DB
- Power Automate
- Power Apps

### 将来的な理想構成

```text
① SharePoint / Box / 社内サーバー
　→ Word、Excel、PDF、過去資料を保存

② 業務ナレッジDB
　→ 業務名、判断基準、注意点、事例を構造化

③ 案件履歴DB
　→ 過去案件、客先、対応履歴、トラブルを蓄積

④ AI検索 / RAG
　→ 「この業務どうやるの？」と聞くと関連資料を探して回答
```

## 17. やってはいけないこと

以下は失敗しやすい。

```text
退職直前に大量の引き継ぎ資料だけ作らせる
フォルダにPDFを置くだけで終わる
誰も更新しないマニュアルを作る
「詳しい人に聞く」で済ませる
AIに全部入れれば何とかなると思う
担当者ごとに資料形式がバラバラ
ファイル名だけで管理する
古い資料と新しい資料を混在させる
責任者を決めずに運用する
```

特に危険なのは、**資料はあるが探せない・古い・信用されていない**状態である。

この状態になると、結局また「詳しい人に聞く」文化に戻る。

## 18. KPI・効果測定

成果測定を決めないまま進めると、「資料は増えたが、実際に良くなったのか」が説明できなくなる。そのため、最初から最低限のKPIを決める。

成果を測る指標としては、以下が使える。

| KPI | 内容 |
|---|---|
| 引継ぎ完了率 | 対象業務のうち、記録・確認済みの割合 |
| 後任の独力対応率 | DBを見て問い合わせなしで完了できた割合 |
| 検索時間 | 必要な資料・判断基準・過去事例を探すのにかかった時間 |
| 問い合わせ件数 | ベテラン・退職者・特定担当者への確認回数 |
| 更新遵守率 | 決められたレビュー期限内に更新・確認された記事の割合 |
| 情報更新率 | 最終更新から3か月以内の記事割合 |
| トラブル復旧時間 | 問題発生時に解決までかかる時間 |
| 属人化業務数 | 特定担当者しかできない業務の件数 |

特に重要なのは、**後任の独力対応率**と**更新遵守率**である。

DBに登録されていても、実際に後任が使えなければ意味がない。  
また、更新期限を過ぎた情報を放置すると信用されなくなるため、未更新記事は管理職または業務オーナーが是正する運用にする。

## 19. 3か月の実行プラン

### 0〜2週目：棚卸し

- 退職予定者ごとの担当業務一覧を作成
- 業務停止時の影響度を確認
- 属人化度を確認
- 優先順位を決める
- テンプレートを確定する

### 3〜6週目：一次収集

- 優先度の高い業務からインタビュー
- 手順、判断基準、失敗例、連絡先を記録
- 関連資料の場所を整理
- 必要に応じて録音・文字起こしを行う

### 7〜10週目：後任による検証

- 後任がDBを見て業務を試す
- 詰まった箇所を追記
- 不足している判断基準を補う
- 古い資料・不足資料を洗い出す

### 11〜12週目：運用化

- 更新責任者を設定
- 月次または四半期レビューを決める
- 退職1か月前の最終確認フローを作る
- 初期KPIを確認する

## 24. 成功条件

この取り組みの成功条件は、以下である。

```text
担当者が退職しても、
過去の経緯・判断基準・手順・注意点・関係者・事例を検索でき、
別の社員が80点レベルで業務を再現できる状態。
```

100点を目指してすべてを完璧に文書化しようとすると、作業量が大きくなりすぎて失敗しやすい。

最初は **80点で再現できる状態** を目指す。

## 26. まとめ

社員の属人的な知識を保存するには、単なるマニュアル化では足りない。

必要なのは、以下を含む業務知識DBである。

```text
手順
判断基準
例外対応
過去トラブル
顧客注意点
関係者
成果物
決定経緯
暗黙知
```

このDBを整備することで、会社は個人の経験に依存した状態から脱却し、退職・異動・世代交代に強い組織へ移行できる。

最初から完璧なシステムを作る必要はない。

まずは、退職したら困る業務を洗い出し、1業務1ページで知識を残すことから始めるべきである。
