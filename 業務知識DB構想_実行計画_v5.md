# 業務知識DB構想・実行計画 v5

## 0. この文書の位置付け

この文書は、以下の内容を統合したものである。

- **DB立案の意図**：なぜ業務知識DBが必要か、何を保存すべきか、最終的にどう使いたいか
- **実行計画**：どの順番で進めるか、どう記録するか、誰が更新するか、どう成果を測るか
- **パイロット計画との接続**：パイロット実施計画（別紙）との関係と位置づけ

本書は、社内説明・初期提案だけでなく、実行時の判断基準・運用設計・将来展開の土台として使うための最終計画書である。  
そのため、短期的なパイロット実施内容だけでなく、Box整理、更新運用、AI活用、ロードマップまでを含めた統合版として位置づける。

### 0.1 本計画で最初に承認を得たいこと

本計画は最終的には全社的な知識資産化を目指すが、最初から大規模展開するものではない。  
まずは営業所単位で小さく始め、実用性・負荷・効果を確認する。

最初に承認を得たい事項は以下である。

```text
1. 営業所単位で業務知識DB構築のパイロットを実施すること
2. 対象営業所・対象業務領域における業務棚卸しを行うこと
3. 若手層への困りごと確認を行うこと
4. ベテラン・有識者への聞き取りを行うこと
5. 高優先業務から週1〜2件程度を目安にDB化すること
6. パイロット対象業務に関連するBoxフォルダを整理対象に含めること
7. パイロット結果をもとに、継続・拡大可否を判断すること
```

### 0.2 本計画で今すぐやること・今すぐやらないこと

本計画は将来的なDB整備・Box整理・AI活用まで含むが、初期段階では範囲を絞る。

| 区分 | 内容 |
|---|---|
| 今すぐやること | 業務棚卸し、優先順位付け、若手への困りごと確認、ベテラン聞き取り、1業務1ページ化、パイロット対象範囲のBox整理 |
| 今すぐやらないこと | 全社一斉展開、大規模システム導入、全Boxの全面整理、AI検索/RAGの本格導入 |
| 将来検討すること | 対象業務の拡大、他部署展開、Box整理範囲の拡大、AI検索/RAG、教育・引継ぎ運用への組込み |

## 1. 目的

社員の頭の中にある業務知識、属人的な判断、過去の経緯、顧客対応の注意点などを会社の資産として保存し、退職・異動・定年によって業務が止まるリスクを減らす。

この取り組みの本質は、単なるマニュアル作成ではない。

**「人の頭の中にある業務の理由・判断・経験を、会社全体で使える知識資産に変えること」** が目的である。

---

## 2. 背景・課題

現在のように、特定の社員に業務知識が集中している状態では、以下のリスクがある。

- 退職・定年で知識が失われる
- 異動時に引き継ぎが追いつかない
- 若手が過去の経緯を知らず、同じ失敗を繰り返す
- 判断基準が人によって違う
- 「詳しい人に聞かないと分からない」状態が続く
- 管理職やベテランが一斉に退職した際、部門全体の判断力が落ちる
- 客先対応や過去案件の注意点が継承されない

特に大きな問題は、手順そのものよりも、**判断基準や過去経緯が失われること**である。

---

## 3. 結論

やるべきことは、広い意味では **業務知識DBの作成** である。

ただし、単なるファイル置き場やExcel管理では不十分である。以下を含む、検索・再利用・更新を前提にした業務知識DBとして整備する必要がある。

- 業務手順
- 判断基準
- 例外対応
- 過去トラブル
- 顧客・取引先ごとの注意点
- 関係者・確認先
- 過去の成果物
- なぜ今の運用になっているのかという経緯
- ベテラン社員の経験則、暗黙知

最終的なイメージは以下である。

```text
社内版Wikipedia
＋
過去案件検索システム
＋
ベテランの判断基準集
＋
AIに聞ける業務相談窓口
```

ただし、AIを活用した検索・相談窓口は最終形であり、最初から導入するものではない。  
まず情報の整理・蓄積を進め、その後にAI活用を検討する。

## 4. この取り組みで重要な考え方

### 4.1 手順書ではなく、判断を残す

通常の引継ぎでは、手順や資料の場所だけが残りがちである。

しかし実際に困るのは、以下のような情報である。

- 例外時にどう判断するか
- 誰に確認すればよいか
- 過去に何で失敗したか
- なぜそのやり方をしているのか
- この客先では何に注意すべきか

そのため、このDBでは単なる作業手順だけでなく、**判断基準・例外対応・過去経緯・失敗知見**を重点的に残す。

### 4.2 完璧ではなく、80点で再現できる状態を目指す

100点を目指してすべてを完璧に文書化しようとすると、作業量が大きくなりすぎて失敗しやすい。

ここでの80点とは、**後任者がDBの内容だけを見て、業務を一通り独力で回せる状態**を指す。  
完璧な手順書ではなく、担当者が変わっても業務が止まらないことが目標である。

### 4.3 最初から大規模にやらない

全業務を一気にDB化しようとすると、負荷が大きくなり途中で止まりやすい。

まずは、退職したら困る業務・過去トラブルが多い業務・客先影響が大きい業務など、危険度の高い業務から小さく始める。

全業務の棚卸しは実施するが、**DB化は優先度の高いものから着手する**。棚卸しとDB化は別の作業として区別する。

### 4.4 パイロットから始める

全社展開・大規模システム導入を前提にせず、まず営業所単位でパイロットを実施する。

パイロットの目的は、**若手が何に困っているかを確認し、ベテランの判断基準を回収し、業務知識DBとして使える形を検証すること**である。

パイロットの詳細は別紙「業務知識DB構築 パイロット実施計画」を参照。

### 4.5 現時点で潰しておくべきリスク

この構想は方向性として妥当だが、実行段階では以下のリスクを先に潰しておく必要がある。

| 指摘・リスク | 具体的に起きる問題 | 先に決めるべき対策 |
|---|---|---|
| 作って終わり化 | 更新責任者・レビュー頻度・承認フローが弱いと、半年程度で情報が陳腐化する | 業務ごとに更新責任者・承認者・レビュー頻度を**最初に**固定する |
| 記述品質のバラつき | 自由記述中心だと、人によって粒度や表現が変わり検索性・再利用性が落ちる | テンプレート・必須項目・記入例・タグ・表記ルールを固定する |
| 優先度の主観化 | 「重要そう」という感覚で始めると工数配分が偏り、重要業務が後回しになる | 優先度スコア = 影響度 + 消失リスク + 代替困難性でスコア化する（各1〜3点、最高9点） |
| 暗黙知の抜け漏れ | ベテラン本人は当たり前を省略しがちで、重要な判断基準や注意点が残らない | インタビュー設問を固定し、判断・例外・失敗・過去経緯を必ず聞く |
| 成果指標の未定義 | 「良くなったか」を定量で示せず、上司・経営層への継続説明が難しくなる | KPIを先に決め、パイロット終了後のアンケートで初期測定する |
| 情報設計の深さ不足 | 顧客別注意点・判断基準・トラブル事例は分類軸を誤ると検索不能になりやすい | タグ設計・分類軸・命名ルールを先に決める |
| 登録は進むが使われない | 利用シーンが未定義だと、登録だけ進んで検索されないDBになる | 誰が・いつ・どの判断で使うかという利用導線を設計する |
| ツール選定先行 | ベンダー比較が先行すると、入力設計・運用設計が弱いままシステム化される | テンプレート・運用ルール・MVPを固めてからツールを選ぶ |

特に重要なのは、**DBを作ること自体を目的にしないこと**である。

## 5. 保存すべき知識の種類

| 種類 | 内容 | 保存方法 |
|---|---|---|
| 手順知識 | どの順番で作業するか | 手順書、チェックリスト |
| 判断知識 | AとBのどちらを選ぶか、なぜそうするか | 判断基準表、ケース集 |
| 例外対応 | 通常と違うケースでどう処理するか | FAQ、トラブル対応DB |
| 顧客・取引先知識 | 客先ごとの癖、注意点、過去経緯 | 顧客別メモ、案件履歴 |
| 人脈・確認先 | 誰に聞けば早いか、どの部署を通すべきか | 関係者マップ |
| 過去経緯 | なぜ今のルール・運用になったか | 決定ログ、議事録、経緯メモ |
| 暗黙知 | ベテランの勘、経験則、注意点 | インタビュー、OJT記録 |
| 成果物知識 | 過去の見積書、提案書、報告書、テンプレート | 成果物DB、ファイル保管庫 |

特に残すべき重要情報は、以下の5つである。

```text
1. なぜそうするのか
2. 何を見て判断するのか
3. どこで失敗しやすいのか
4. 誰に確認すべきなのか
5. 過去に何があったのか
```

## 6. 業務知識DBの構成案

### 6.1 業務手順DB
業務ごとの基本的な手順を保存する。

### 6.2 判断基準DB
担当者の経験に依存している判断を保存する。

### 6.3 トラブル事例DB
過去に起きた失敗やクレーム、再発防止策を保存する。

### 6.4 顧客・取引先注意点DB
客先や取引先ごとの注意事項を保存する。

### 6.5 人材・スキルDB
社内で誰が何に詳しいかを整理する。「誰に聞けばよいか」が明確になる。  
※人員変動により情報が陳腐化しやすいため、定期的なメンテナンスが必要。

### 6.6 成果物DB
過去の資料や成果物を検索できるようにする。

### 6.7 決定経緯DB
なぜ現在のルールや運用になったのかを保存する。これがないと、後任者が同じ問題を繰り返す可能性がある。

## 7. 優先順位の付け方

全業務を一度に整備するのではなく、危険度の高いものから着手する。

優先順位は以下の考え方で決める。

```text
優先度スコア = 影響度 + 消失リスク + 代替困難性
```

各指標は以下の基準でスコア化する。合計が高い業務から優先的にDB化する（最高9点）。

| 指標 | 1点 | 2点 | 3点 |
|---|---|---|---|
| 影響度 | 業務が止まっても影響が小さい | 部門内に影響が出る | 客先・会社全体に影響が出る |
| 消失リスク | 複数名が知っている | 担当者が限られる | 1名しか知らない |
| 代替困難性 | 誰でもすぐ対応できる | 習得に時間がかかる | 代替が事実上不可能 |

スコアが高い業務から優先的にDB化する。

### 7.1 優先順位スコアの補助ルール

合計点は優先順位を決めるための基本指標である。  
ただし、点数だけで機械的に判断すると、重要業務を見落とす可能性があるため、以下の補助ルールを設ける。

| 補助ルール | 扱い |
|---|---|
| 影響度が3点の業務 | 合計点が低くても優先確認対象とする |
| 消失リスクが3点かつ代替困難性が3点の業務 | 原則として優先DB化対象とする |
| 顧客対応・金額判断・法務/契約に関わる業務 | 関係者確認を行い、必要に応じて優先度を上げる |
| 若手から困りごとが多く挙がった業務 | スコアに加えて実務上の困り度を考慮する |
| 既存資料が古い、または最新版が不明な業務 | Box整理とあわせて優先確認対象とする |

スコアはあくまで判断材料であり、最終的な着手順は、業務影響・知識消失リスク・現場の困りごとを総合して決める。

この管理表は**DB化の前に作成する**。棚卸しとDB化は別の作業であり、棚卸しで優先度を確認してからDB化に着手する。

| 業務名 | 現担当者 | 代替可能者 | 属人化度 | 失われた場合の影響 | 優先スコア（最高9点） | 備考 |

| 月次処理 | Dさん | Eさん | 中 | 中 | 6（2+2+2） | 手順書はあるが古い |
| システム入力 | Fさん | なし | 高 | 中 | 7（3+2+2） | 例外処理が未整理 |

### 9.1 記述標準化のためのルール

| ルール | 内容 |
|---|---|
| 必須項目を固定する | 目的・手順・判断基準・例外対応・失敗例・関連資料・更新情報は必ず記入する |
| 不明点を空欄にしない | 分からない場合は「不明」「要確認」と明記する |
| 判断理由を書く | 「何をするか」だけでなく「なぜそうするか」を書く |
| 1記事1業務にする | 複数業務を1ページに混ぜない |
| タグを必ず付ける | 部署・業務名・頻度・重要度・情報種別・公開範囲を付ける |
| 更新情報を残す | 最終更新日・更新者・責任者・次回見直し日を記録する |

## 10. ベテラン社員への聞き取り項目

ベテラン社員には手順書を書いてもらうより、まずインタビューする方がよい。  
自由記述にすると、本人にとって当たり前すぎる重要情報が抜けやすい。質問形式で判断・失敗・例外・経緯を引き出す。

```text
この業務で新人が一番ミスるところは？
過去に一番大きく揉めた案件は？
この業務で絶対に確認すべき相手は？
資料には書いていないが、実際には気をつけていることは？
この判断を間違えると何が起きる？
あなたが退職したら一番困る業務は？
この業務で昔と今で変わった点は？
なぜ今のやり方になったのか？
この客先・取引先で注意すべきことは？
過去資料を見るならどのフォルダ・どの案件を見るべきか？
トラブル時に最初に見る数字・資料は？
後任が一番困りそうな判断は？
これはやってはいけない、ということは？
```

## 11. 導入ステップ

### Step 1：運用ルール・更新責任者を先に決める

DB化を始める前に、以下を固定する。

- 誰が更新するか（業務ごとに更新責任者を1名決める）
- 誰が内容を確認・承認するか
- いつ見直すか（最低でも四半期に1回）
- 承認フローの状態区分（下書き / レビュー中 / 承認済み / 要更新 / 廃止）
- 廃止情報の扱い（アーカイブ領域に移動し、現行資料と混在させない）

**更新責任者の設定はDB化の着手前に行う。後から決めると「作って終わり」になりやすい。**

### Step 2：パイロット対象業務の選定

まず、感覚的に「これが危ない」と判断できる業務・業務領域を1〜2件選ぶ。  
この段階では全業務を網羅する必要はない。「退職したら止まる」「若手が毎回困っている」業務を起点に選ぶ。

全業務の棚卸しはパイロット後に実施する。先に全業務を洗い出そうとすると初動の負荷が大きくなり、着手前に止まりやすい。

### Step 3：パイロット実施（別紙参照）

営業所単位で小規模なパイロットを実施し、若手の困りごとを起点にベテランの知識を回収する。  
詳細は別紙「業務知識DB構築 パイロット実施計画」を参照。

パイロットを先行させることで、以下が得られる。

- 「何を確認すべきか」「どういう形式で整理すればよいか」が分かった状態で棚卸しに入れる
- 実際の工数感・負荷・使いやすさが確認でき、全体棚卸しの設計に活かせる
- 棚卸し前に現場への説明材料が揃う

### Step 4：全業務の棚卸し・優先順位付け

パイロットで得た知見をもとに、全業務の一覧を作成し属人化リスクを可視化する。  
この段階では整理するだけであり、DB化は次以降のステップで行う。

- どの業務があるか
- 誰が担当しているか
- 代替できる人がいるか
- 属人化しているか
- 失われた場合の影響はどれくらいか

セクション7の優先スコアをもとに、DB化する業務の順番を決める。

### Step 5：1業務1ページに整理する

聞き取った内容をテンプレートに沿って整理する。  
最初から完璧を目指さず、後任者が独力で業務を回せる状態（80点）を目標にする。

### Step 6：実案件で検証する

別の社員が、その資料だけを見て実際に業務を進められるか確認する。  
詰まった箇所が、知識の抜けである。

### Step 7：AI検索を導入する

資料がある程度整理されてから、AI検索やRAGを導入する。  
元資料が整理されていない状態でAIを導入しても役に立たない。

## 12. Box整理との連携

### 12.1 位置づけ

現状のBoxはデータが大量にあるが、構造が分かりにくい状態にある。  
これをそのままにしてDBだけ作っても、「Boxの中に答えがあるのに探せない」状態が解消されない。

そのため、**Box整理はDB構築の前処理**として位置づけ、パイロット対象業務と並行して実施する。  
ここでいうBox整理は、単なるフォルダ美化ではなく、業務知識DBの根拠資料・最新版資料・過去経緯を探せるようにするための基盤整備である。

### 12.2 進め方

全体のBox整理は工数が大きすぎるため、一度に手をつけない。  
**パイロット対象業務に関連するBoxフォルダだけを先行して整理する**。

整理する内容は以下とする。

- この業務に関連する資料がどこにあるか
- 最新版とそれ以外の区別
- 各資料がいつ・何のために使うものか
- 古い資料・廃止資料はアーカイブフォルダに移動

この整理ノウハウを、後から他部署・他業務への展開に活用する。  
また、将来的にAI検索やRAGを導入する場合も、Box内資料の構造・命名・最新版管理が整理されていることが前提になるため、パイロット段階から最低限の整理ルールを確認しておく。

### 12.3 AIを使った整理について

Box内資料の分類・命名・重複検出にAIを補助的に活用することで、純粋な人力より効率化できる可能性がある。

ただし以下の点を先に確認する必要がある。

- 社内の情報管理ルール上、Box資料をAIに読み込ませることが可能か
- クラウドAIに送れる情報の範囲はどこまでか

**AI活用のための予算はパイロット終了後に申請する**。  
パイロット実施により「Box整理に要した工数」と「AI活用による削減効果の見込み」を数字で示してから予算申請することで、承認を得やすくなる。

### 12.4 Box整理の最低ルール

パイロット対象業務に関連するBoxフォルダでは、最低限以下を確認する。  
ここでの目的は、フォルダを見た目だけきれいにすることではなく、業務知識DBから根拠資料・最新版資料・過去経緯にたどれる状態を作ることである。

| 確認項目 | 整理の基準 |
|---|---|
| 最新版資料の判別 | 現行で使う資料が分かる状態にする |
| 過去資料・廃止資料の分離 | 現行資料と混在させず、アーカイブ領域へ移動する |
| 関連資料へのリンク | 業務知識DBの記事から該当資料へたどれるようにする |
| ファイル名の分かりやすさ | ファイル名だけで内容・用途がある程度分かる状態にする |
| 用途不明資料の扱い | 削除せず「要確認」として記録し、関係者確認の対象にする |
| 資料の使用タイミング | どの業務・どの判断で使う資料かをメモする |

Box整理の完了条件は、対象業務の記事から必要資料へ到達でき、最新版・過去資料・廃止資料の区別ができる状態とする。

## 13. MVP案：最小構成で始める

### 初期対象

```text
対象範囲：属人化リスクが高い営業所・業務領域を1つ選ぶ（パイロット対象）
対象業務：まず10件程度を上限とする
対象者：ベテラン社員・有識者・業務が属人化している担当者・後任候補者
```

※退職・異動・担当変更により知識消失リスクが高い業務がある場合は、優先的に対象へ含める。

### 収集方法

```text
1業務あたり：
- 事前準備・関連資料の確認
- インタビュー（60分程度）
- 内容整理・初稿作成
- レビュー・修正
- 後任候補者による実行確認

目安工数：1業務あたり2〜3時間程度
```

単純な手順書作成ではなく、判断基準・例外対応・過去経緯・関連資料まで整理するため、1業務あたりの工数は余裕をもって見積もる。

### 登録ペース

通常業務と並行するため、週あたりの工数を現実的に見積もってから着手ペースを決める。  
初期段階では**週1〜2件程度**を目安とする。ただし、対象業務の複雑さやインタビュー対象者の都合により調整する。

### 13.1 パイロット段階で使用するツール

パイロット段階では、新規ツールの導入や追加コストを発生させず、既存環境で動かすことを優先する。

```text
1業務1ページの管理：Excel（1シート＝1業務のテンプレート形式）
関連資料の参照：Box（整理済みフォルダへのリンクをExcelに記載）
アンケート集計：Excel
```

この構成であれば社内承認・追加費用なしでパイロットを開始できる。

パイロットでは業務知識の収集・検証だけでなく、**Excelによる管理がどの件数・規模まで現実的に機能するか**も確認対象とする。

具体的に確認したいのは以下の点である。

| 確認項目 | 判断基準 |
|---|---|
| 検索性 | 件数が増えたときに必要な記事を素早く探せるか |
| 更新管理 | 誰がいつ更新したかを追跡できるか |
| 共有・アクセス管理 | 複数人での参照・編集に耐えられるか |
| リンク管理 | Box資料へのリンクが維持・管理できるか |

パイロット終了後、上記の確認結果をもとにツール移行の必要性を判断する。  
限界が確認された場合は、SharePointまたはMicrosoft Listsへの移行を提言する。  
この提言には「Excelで〇件管理した結果、具体的にここが限界だった」という実績を根拠として添える。

### 13.2 実施体制・役割分担

パイロットを実行する際は、役割を曖昧にしない。  
特に、記録する人、内容を確認する人、実際に使って検証する人を分けることで、作って終わりになることを防ぐ。

| 役割 | 主な担当内容 |
|---|---|
| 推進担当 | 全体進行、対象業務の管理、棚卸し表の更新、進捗確認 |
| 業務担当者 | 対象業務の説明、関連資料の提示、初稿内容の確認 |
| ベテラン・有識者 | 判断基準、例外対応、過去経緯、失敗事例の提供 |
| 後任候補者・若手 | DBを見て業務が進められるか検証し、不足情報を指摘する |
| 承認者 | 内容の正確性・運用上の妥当性を確認し、承認判断を行う |
| Box整理担当 | 関連資料の所在確認、最新版・過去資料・廃止資料の整理を行う |

小規模パイロットでは、1人が複数の役割を兼ねてもよい。  
ただし、最低限「作成者」と「確認者」は分ける。特に承認済み記事にする前には、業務担当者または有識者による確認を必須とする。

## 14. 運用ルール

業務知識DBとして運用するなら、登録だけでなく更新・承認・検索のルールが必要である。  
ここが曖昧だと、DBは作って終わりになり、数か月後には古い情報の置き場になる。

| 項目 | 内容 |
|---|---|
| 更新責任者 | 業務ごとに記事を更新する担当者を1名決める（DB化着手前に設定） |
| 承認者 | 業務内容の正しさを確認する管理職または有識者を決める |
| 代理担当 | 責任者不在時に確認・更新できる人を決める |
| 更新頻度 | 最低でも四半期に1回見直す。重要業務は月次でもよい |
| 承認フロー | 下書き → レビュー中 → 承認済み → 要更新、の状態を分ける |
| 廃止資料の扱い | 廃止状態の資料は**アーカイブ領域に移動**し、現行資料と混在させない |
| タグ設計 | 部署・業務名・システム・頻度・重要度などを統一する |
| 権限管理 | 全体公開・部門限定・機微情報に分ける |
| 改訂履歴 | いつ・誰が・何を変更したかを残す |
| 完了定義 | 後任が見て独力で業務を実行できたら完了とする |

### 14.1 承認フローの状態区分

```text
下書き：聞き取り内容を整理した段階。未確認情報を含む。
レビュー中：後任者・管理職・有識者が内容を確認している段階。
承認済み：業務手順として参照してよい状態。
要更新：内容が古い、または実務とズレている可能性がある状態。
廃止（アーカイブ）：現在は使わないが過去経緯として保管する状態。
　→ アーカイブ領域に移動し、現行ページとは分離して管理する。
```

### 14.2 利用導線の設計

DBは登録されているだけでは使われない。  
誰が・いつ・どの場面で参照するのかを先に決め、実務の流れに組み込む必要がある。

| 利用場面 | 使う人 | 参照する情報 |
|---|---|---|
| 退職・異動時の引継ぎ | 後任者・管理職 | 業務手順・判断基準・関連資料・確認先 |
| 新人・若手教育 | 新人・教育担当 | 基本手順・よくある失敗・成果物サンプル |
| トラブル発生時 | 担当者・管理職 | 過去トラブル事例・例外対応・確認先 |
| 顧客対応前 | 営業・担当者 | 顧客別注意点・過去経緯・類似案件 |
| 業務改善時 | 管理職・担当者 | 決定経緯・過去の廃止ルール・失敗事例 |

## 15. 検索性とタグ設計

DB化する以上、検索できることが重要である。  
記事や資料には最低限以下のタグを付ける。

| タグ | 例 |
|---|---|
| 部署 | 営業・技術・総務・経理 |
| 業務名 | 見積作成・客先対応・月次処理 |
| システム | 社内システム名・Excel・Box・SharePoint |
| 頻度 | 毎日・毎月・年次・随時 |
| 重要度 | 高・中・低 |
| 属人化度 | 高・中・低 |
| 顧客・取引先 | 顧客名・取引先名 |
| 情報種別 | 手順・判断基準・トラブル・成果物・経緯 |
| 公開範囲 | 全体公開・部門限定・機微情報 |
| ステータス | 承認済み・レビュー中・要更新・廃止（アーカイブ） |

## 16. AI活用方針

AIは、知識保存の中心ではなく、**整理・検索・初期作成の補助**として使う。

### AIに向いていること

- 録音したインタビューの文字起こし
- 長い説明の要約
- 手順書の初版作成
- FAQ化
- Box内資料の分類・命名候補の提示
- 類似案件の検索
- 資料から注意点を抽出
- 新人向け説明文の作成

### AIに任せきりにしてはいけないこと

- 最終的な業務判断
- 社内ルールの確定
- 法務・契約・金額判断
- 顧客対応方針の決定
- 古い資料と新しい資料の正誤判定

AIは便利だが、元データが間違っていれば回答も間違う。  
そのため、AI導入前に**資料の整理・更新ルール・責任者設定**が先決である。

## 17. KPI・効果測定

成果測定を決めないまま進めると、「資料は増えたが実際に良くなったのか」が説明できなくなる。

### 17.1 測定方法

定量的なKPI値は、パイロット終了後に若手・後任者へのアンケートで初期測定する。

アンケートの流れは以下とする。

```text
1. 若手層に業務上の困りごとを確認（パイロット前）
2. 上位層・ベテランへ内容を共有し、不足知識を回収
3. DB作成後、若手層へフィードバックアンケートを実施
　→「必要な情報が見つかるか」「判断基準が分かるか」「以前より困りごとが減ったか」を確認
```

### 17.2 KPI一覧

| KPI | 内容 |
|---|---|
| 後任の独力対応率 | DBを見て問い合わせなしで完了できた割合 |
| 若手の困りごと解消率 | パイロット前後のアンケート比較で改善を確認 |
| 引継ぎ完了率 | 対象業務のうち、記録・確認済みの割合 |
| 検索時間 | 必要な資料・判断基準を探すのにかかった時間 |
| 問い合わせ件数 | ベテラン・退職者・特定担当者への確認回数 |
| 更新遵守率 | 決められたレビュー期限内に更新・確認された記事の割合 |
| 属人化業務数 | 特定担当者しかできない業務の件数 |

特に重要なのは**後任の独力対応率**と**更新遵守率**である。

### 17.3 パイロット終了時の判定基準

パイロット終了時には、単に記事数だけで評価しない。  
以下の観点から、継続・拡大可否を判断する。

| 判定項目 | 確認内容 |
|---|---|
| DB化の進捗 | 対象業務10件程度のうち、何件を承認済みにできたか |
| 後任の独力対応 | 後任候補者・若手がDBを見て業務を進められたか |
| 困りごとの解消 | パイロット前に出た若手の困りごとがどの程度解消されたか |
| 問い合わせの変化 | ベテラン・特定担当者への確認回数が減ったか |
| Box整理の効果 | 関連資料について、最新版・過去資料・廃止資料の区別ができたか |
| 工数の妥当性 | 通常業務と並行して継続可能な負荷だったか |
| 横展開の可能性 | 他業務・他営業所へ展開する際の課題が明確になったか |

パイロットの成功条件は、対象業務すべてを完璧にDB化することではない。  
重要なのは、業務知識DBが実務で使えるか、どの運用ルールが必要か、どの程度の工数がかかるかを確認できることである。

## 18. やってはいけないこと

```text
退職直前に大量の引き継ぎ資料だけ作らせる
フォルダにPDFを置くだけで終わる
誰も更新しないマニュアルを作る（更新責任者は最初に決める）
「詳しい人に聞く」で済ませる
AIに全部入れれば何とかなると思う
担当者ごとに資料形式がバラバラ
ファイル名だけで管理する
廃止資料と現行資料を同じ場所に混在させる
責任者を決めずに運用する
ツール選定を先行させる
```

特に危険なのは、**資料はあるが探せない・古い・信用されていない**状態である。  
この状態になると、結局また「詳しい人に聞く」文化に戻る。

## 19. ロードマップ（2年スパン）

| 期間 | 内容 |
|---|---|
| 0〜1か月 | 運用ルール・更新責任者の設定。テンプレート確定。危険度の高い業務をざっくり選定してパイロット開始準備。 |
| 1〜3か月 | パイロット実施（営業所単位）。若手ニーズ確認・ベテラン知識回収・DBサンプル作成・検証。パイロット対象Boxフォルダの整理。 |
| 4〜6か月 | パイロット結果の整理。**全業務の棚卸し・優先順位付け**（パイロットの知見を活用）。対象業務を拡大するか判断。 |
| 7〜12か月 | 高優先業務を中心にDB化を継続。属人化リスクが高い業務・引継ぎリスクが高い業務の知見を重点的に回収。 |
| 1年〜1年半 | 更新フロー・レビュー体制・品質確認を定着。他部署への展開検討。 |
| 1年半〜2年 | Box整理の範囲拡大。AI活用予算の申請・ツール選定。新人教育・異動時引継ぎへの組込み。 |

## 20. 推奨する保存先・ツール構成

ツール選定はロードマップの後半で行う。先行させない。

順番は以下が望ましい。

```text
1. 何を残すかを決める
2. どのテンプレートで記録するかを決める
3. 誰が更新・承認するかを決める
4. どう検索するか、タグを決める
5. その運用に合うツールを選ぶ
```

### 小規模開始

- Excel
- SharePoint
- Microsoft Lists
- 社内共有フォルダ
- OneNote / Notion / Confluence

### 将来的な理想構成

```text
① SharePoint / Box / 社内サーバー
　→ Word・Excel・PDF・過去資料を保存

② 業務ナレッジDB
　→ 業務名・判断基準・注意点・事例を構造化

③ 案件履歴DB
　→ 過去案件・客先・対応履歴・トラブルを蓄積

④ AI検索 / RAG
　→ 「この業務どうやるの？」と聞くと関連資料を探して回答
　→ Box整理・DB整備が一定水準に達した後に導入
```

## 21. 成功条件

```text
担当者が退職しても、
過去の経緯・判断基準・手順・注意点・関係者・事例を検索でき、
後任者が独力で業務を一通り回せる状態。
```

100点を目指してすべてを完璧に文書化しようとすると、作業量が大きくなりすぎて失敗しやすい。  
最初は**後任者が独力で業務を回せる状態（80点）**を目指す。

## 22. まず次にやるべきこと

```text
1. 運用ルール・更新責任者を決める
2. 推進担当・確認者・承認者などの実施体制を決める
3. 属人化リスクが高い営業所・業務領域を1つ選ぶ
4. 危険度が高い業務を感覚的に数件選び、パイロットを開始する
5. パイロット対象業務に関連するBoxフォルダの整理範囲を決める
6. パイロット終了後に全業務の棚卸しを行い、属人化リスクを可視化する
7. 棚卸し結果をもとに高優先業務を選定し、DB化を継続する
8. パイロット終了時の判定基準をもとに、継続・拡大可否を判断する
```

※具体的な担当者名・工数配分・実施時期は、社内確認後に決定する。

## 23. まとめ

社員の属人的な知識を保存するには、単なるマニュアル化では足りない。

必要なのは以下を含む業務知識DBである。

```text
手順 / 判断基準 / 例外対応 / 過去トラブル
顧客注意点 / 関係者 / 成果物 / 決定経緯 / 暗黙知
```

このDBを整備することで、会社は個人の経験に依存した状態から脱却し、退職・異動・世代交代に強い組織へ移行できる。

最初から完璧なシステムを作る必要はない。  
まずは更新ルールと責任者を決め、属人化リスクが高い業務を洗い出し、パイロットから小さく始める。
